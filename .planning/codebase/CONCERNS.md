## Overview

This is an early-stage hackathon project (YHack 2026) with multiple communication approaches (BLE, Serial, simulated) that have not been unified. The web frontend is functional but entirely untracked in git, API keys are exposed client-side by design, CORS is wide open everywhere, and there are zero tests across the entire codebase. The project works for demo purposes but has significant security, reliability, and maintainability gaps.

## Critical Concerns

### API Keys Exposed in Browser JavaScript

- **Issue:** Both the Anthropic API key and ElevenLabs API key are accessed via `NEXT_PUBLIC_` environment variables, meaning they are bundled into client-side JavaScript and sent to the browser. Anyone inspecting the page source can extract these keys.
- **Files:**
  - `web/hooks/usePostureSession.ts` (line 187) -- calls `https://api.anthropic.com/v1/messages` directly from the browser with `"anthropic-dangerous-direct-browser-access": "true"` header (line 202)
  - `web/hooks/useVoiceAlert.ts` (line 17) -- calls `https://api.elevenlabs.io/v1/text-to-speech/` directly from the browser
  - `web/.env.example` (lines 6-12) -- documents both as `NEXT_PUBLIC_` vars
- **Impact:** Anyone who loads the page can steal API keys and run up charges. The `anthropic-dangerous-direct-browser-access` header is explicitly labeled dangerous.
- **Fix approach:** Create backend proxy endpoints on the FastAPI server (`/api/coach-tip`, `/api/voice-alert`) that hold keys server-side. Frontend calls those proxies instead.

### CORS Allows All Origins on All Servers

- **Issue:** Every Python server uses `allow_origins=["*"]` with all methods and all headers, which means any website on the internet can make requests to these endpoints.
- **Files:**
  - `web/server/main.py` (line 25): `allow_origins=["*"]`
  - `serial/posture_detector.py` (line 126): `allow_origins=["*"]`
  - `ble/reader.py` (line 134): `allow_origins=["*"]`
- **Impact:** If the server is exposed beyond localhost (the servers bind `0.0.0.0`), any external site can connect to the WebSocket and read posture data or issue calibration commands.
- **Fix approach:** Restrict `allow_origins` to `["http://localhost:3000"]` or the actual frontend domain. If needed for hackathon flexibility, at least restrict in production.

### Servers Bind to 0.0.0.0 (All Network Interfaces)

- **Issue:** Both `serial/posture_detector.py` (line 220) and `ble/reader.py` (line 184) bind to `0.0.0.0`, exposing the API to the entire local network. Combined with wildcard CORS and no authentication, anyone on the same WiFi can read sensor data or send commands.
- **Files:** `serial/posture_detector.py` (line 220), `ble/reader.py` (line 184)
- **Impact:** At a hackathon with shared WiFi, other teams could interfere with the demo.
- **Fix approach:** Bind to `127.0.0.1` unless LAN access is explicitly needed.

## Technical Debt

### Three Duplicate Server Implementations

- **Issue:** There are three separate FastAPI servers that do largely the same thing (serve posture data over WebSocket/REST), each with different feature sets and slightly different APIs:
  1. `web/server/main.py` -- Simulation-only, WebSocket at `/ws`, sends `{delta, timestamp}`
  2. `serial/posture_detector.py` -- Real serial hardware, WebSocket at `/ws` + REST at `/tilt`, sends `{delta, timestamp}` over WS and `{pitch, roll, posture, timestamp}` over REST
  3. `ble/reader.py` -- BLE hardware, REST at `/tilt` only (no WebSocket), sends `{pitch, roll, posture, timestamp}`
- **Impact:** Feature drift between implementations. The web frontend only uses WebSocket, so the BLE reader is incompatible. Bug fixes need to be applied in three places.
- **Fix approach:** Unify into a single server with pluggable data sources (simulator, serial, BLE) selected by command-line flag or environment variable.

### Memory Leak: Session Data Grows Unbounded

- **Issue:** `allSessionDataRef` in `usePostureSession.ts` appends one data point per second forever. For a long session, this array grows without limit. After 8 hours that would be ~28,800 data points, which is manageable, but there is no cap or cleanup.
- **Files:** `web/hooks/usePostureSession.ts` (lines 33, 143)
- **Impact:** Memory usage grows linearly with session duration. For a hackathon demo this is acceptable, but for production use it would cause browser sluggishness over extended periods.
- **Fix approach:** Cap `allSessionDataRef` at a reasonable maximum (e.g., 3600 entries = 1 hour) and shift old entries out. The `recentChartData` view already slices to the last 600 entries.

### Global Mutable State in Python Servers

- **Issue:** Both `serial/posture_detector.py` and `ble/reader.py` use module-level global variables (`latest_reading`, `baseline_pitch`, `baseline_roll`) shared between threads via implicit mutation without locks.
- **Files:**
  - `serial/posture_detector.py` (lines 25-37, 77, 101, 153-154, 170-171)
  - `ble/reader.py` (lines 38-43, 55, 65)
- **Impact:** Potential race conditions: the serial thread writes `latest_reading` while the async WebSocket handler reads it. In CPython the GIL makes dict replacement atomic, but this is an implementation detail, not a guarantee.
- **Fix approach:** Use `threading.Lock` to protect shared state, or use `asyncio.Queue` to pass data from the serial/BLE thread to the async event loop.

### Simulation Server Shares One Simulator Across All Clients

- **Issue:** `web/server/main.py` uses a single global `IMUSimulator` instance (line 115). All connected WebSocket clients see the same simulated posture data. If one client sends a calibrate command, all clients are affected.
- **Files:** `web/server/main.py` (line 115, line 157)
- **Impact:** Unexpected behavior with multiple browser tabs or concurrent users.
- **Fix approach:** Create a per-connection `IMUSimulator` instance inside `websocket_endpoint()`.

### @types/react Version Mismatch

- **Issue:** `package.json` specifies `react: ^19.2.4` but `@types/react: ^18`. React 19 has breaking type changes (e.g., no implicit `children` on `FC`, new `use()` hook types). TypeScript compilation uses `skipLibCheck: true` which masks these mismatches.
- **Files:** `web/package.json` (lines 14-15 vs line 19), `web/tsconfig.json` (line 9)
- **Impact:** Type errors are silently suppressed. Developers lose type safety for React 19 features.
- **Fix approach:** Upgrade `@types/react` and `@types/react-dom` to `^19`.

## Missing Pieces

- **Files:** No `*.test.*` or `*.spec.*` files exist outside `node_modules/`

### No Error Feedback to Users

- **Issue:** All `catch` blocks in the frontend silently swallow errors. WebSocket parse errors, API call failures (Anthropic, ElevenLabs), and audio playback failures are all caught and ignored.
- **Files:**

  - `web/hooks/usePostureSession.ts` (line 220): catch on Anthropic API call

- **Impact:** When the AI coach or voice alerts fail, the user gets no indication. The tip just shows the default text. Debugging requires browser DevTools.
- **Fix approach:** Add a toast/notification system for transient errors. At minimum, log errors to console in development.

### No Persistent Session Storage

- **Issue:** All session data (stats, chart history, best streak) lives in React state and refs. Refreshing the page or navigating away loses everything.
- **Files:** `web/hooks/usePostureSession.ts` (entire file)
- **Impact:** Users lose their posture history on page refresh. No ability to track posture over days/weeks.
- **Fix approach:** Persist session summaries to `localStorage` or IndexedDB. For multi-device support, add a database backend.

### No Authentication or User Identity

- **Issue:** No concept of users, sessions, or authentication. The WebSocket server accepts all connections. The API endpoints are open.
- **Files:** All server files
